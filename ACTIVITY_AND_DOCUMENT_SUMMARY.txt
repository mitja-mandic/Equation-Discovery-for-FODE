WSL2 SECURITY AND WORK SUMMARY

Security work completed before 13:00 on 14 July 2026:

The environment was assessed as a WSL2 Linux guest running under Windows. The goal was to allow the Linux account to administer the guest freely, while removing the usual WSL2 convenience paths through which a compromised guest account can reach Windows files or launch Windows programs.

The active /etc/wsl.conf configuration applies the following controls:
- Windows interoperation is disabled (interop.enabled=false). Linux processes can no longer directly start Windows executables through the normal WSL interop mechanism.
- The Windows PATH is not imported into Linux (appendWindowsPath=false). This prevents Windows program directories from being automatically placed on the Linux command path.
- Automatic Windows-drive mounting is disabled (automount.enabled=false). Drives such as C: are not automatically exposed below /mnt/c, so ordinary Linux processes do not have direct access to host files through the standard DrvFS mount.
- /etc/fstab processing is disabled for WSL startup (mountFsTab=false). This reduces the chance that a later fstab entry silently restores a host filesystem mount.
- systemd is enabled and the default WSL user is mitjam, preserving a normal Linux service environment and predictable login identity.

Passwordless administration was then configured inside the guest only. The helper script created /etc/sudoers.d/mitjam with a narrowly scoped sudoers entry for the mitjam account, set the required 0440 permissions, and validated it with visudo when available. This allows the account to use sudo without a password for Linux guest administration; it does not grant Windows or host-administrator privileges.

The configuration was verified after application. The active /etc/wsl.conf matches the hardening configuration above, and no Windows DrvFS mounts such as /mnt/c are present. WSL-owned runtime mounts (for example /mnt/wsl and WSLg support mounts) remain because they are managed by the WSL platform rather than being Windows-drive mounts.

Security boundary and operating note:
These measures significantly reduce common guest-to-host file and command execution paths, but WSL2 is not a complete hostile-workload sandbox. The Windows host and its WSL management layer remain more privileged than the guest. Future workflows that need files moved from Windows into the guest should use an explicit, controlled transfer mechanism instead of re-enabling interop or automatic drive mounting.
